# 코드·비즈니스 로직 분석 (code-analyze-module)

사용자가 지정한 (기능명 또는 파일명을 주입) **진입점**(파일·폴더·기능 설명)을 중심으로 복잡한 모듈 또는 비즈니스 로직을 읽고, 아래 **산출 마크다운 구조**로 정리한다. 저장소 전체 맵이 아니라 **해당 범위만** 다룬다.

산출물은 **코드를 처음 읽는 사람이 인지하는 순서**(디렉토리 지도 → 진입 → 실행 → 책임·규칙 → 훅·모듈 연결)에 맞춘다.

같은 팀/저장소에 이미 비슷한 분석 문서가 있으면 **목차·절 구조만** 참고하고, 내용은 이번 진입점에 맞게 새로 쓴다.

---

## 에이전트 동작 규칙

1. **Read Before Touch**: 진입점과 직접 연결된 파일(의존성·호출)을 읽은 뒤에만 서술한다. **추측은 기본적으로 쓰지 않는다.** 불가피할 때만 `추정:`으로 표시하고, 근거가 부족한 이유를 한 줄 덧붙인다.
2. **Spec First (분석)**: 이 범위와 관련된 **PRD/TRD/설계·요구 문서**가 저장소(또는 팀이 지정한 SSOT 경로)에 있으면 **분석 전에 확인**하고, 산출물의 `## 분석 목적과 접근`에 **한두 문장 또는 근거 링크**로 반영한다. 해당 문서가 없거나 범위 밖이면 **`해당 없음`**으로 명시한다.
3. **진입점**: 사용자 메시지의 경로, 에디터에서 지정한 파일 참조, 또는 질문으로 확정한다. (필수)
4. **범위·과대 진입점**: 한 번에 읽기 어려울 만큼 넓으면 **하위 모듈·기능 단위로 나누어 분석**하거나, 사용자에게 **범위 축소·우선 진입점**을 질문해 확정한다. (모호할 때는 질문 우선)
5. **출력 위치**(선택): 사용자가 지정한 경로에 저장하거나, 채팅에만 마크다운을 출력한다.
6. **팀 규칙**: 이 저장소에 `README`, `CONTRIBUTING`, `docs/`, `rules/` 등 **코딩·아키텍처 가이드**가 있으면 준수한다. 검증 책임 위치·오류 처리 방식(예: 진입부 검증, 실패 시 명시적 전파)이 문서화되어 있으면 한두 문장으로만 반영한다. **저장소마다 용어가 다르므로** 특정 파일명을 전제로 하지 않는다.

---

## 본 지침 파일 참조 (링크)

이 커맨드 문서를 **지칭·인용·안내**할 때는 경로와 파일명이 **한 번에 열리도록** 마크다운 링크로 쓴다. 파일명만 단독으로 쓰거나 백틱만 쓰지 않는다.

- **형식**: `[표시 이름](저장소 루트 기준 상대 경로)`
- **고정 예시**: [`devoks-code-analyze-module`](.claude/commands/devoks-code-analyze-module.md)
- **표시 이름**: `code-analyze-module` 또는 `코드·비즈니스 로직 분석 (code-analyze-module)` 등 맥락에 맞게 선택한다.

PR 설명, 이슈, 다른 `.md` 규칙/문서, 채팅에서 이 지침을 가리킬 때 위 링크 형식을 우선한다.

---

## Mermaid (간단 규칙)

- 노드 ID에는 **공백 금지** (camelCase, PascalCase, underscores).
- 라벨에 괄호·콜론 등이 있으면 노드에 `"double quotes"` 사용.
- 예약어 `end`, `subgraph` 등을 노드 ID로 쓰지 않는다.
- 다이어그램은 **3개 이내** 권장: **§5** 책임 레이어 1, **§4** 시퀀스 1, **§6** 정적 의존 1. 필요 시 하위 모듈만 추가 서브그래프 1개.
- **정적 의존** 다이어그램은 산출 템플릿에서 **§6** 안에만 둔다(Provider·훅·모듈 연결 맵을 본 뒤). **§5와 중복 배치하지 않는다.**

## 분석 산출물의 Definition of Done (DoD)

워크스페이스 **완료 기준**과 맞추되, **코드 분석 문서**에 한정한다.

- **Spec**: 관련 PRD/TRD·요구 문서를 확인했으면 산출물에 반영했는가. 없으면 **`해당 없음`**을 적었는가.
- **코드 일치**: 서술이 읽은 코드·계약과 맞는가. 스펙과 코드가 다르면 **코드 동작을 기준**으로 쓰고, 스펙과의 차이를 **`추정:` 없이** 사실만 짚었는가(있을 때만).
- **테스트**: 관련 단위·통합 테스트 파일이 **식별 가능하면** 산출물(`## 분석 목적과 접근`, **§6**, 또는 **부록 A**)에 **경로 또는 요약**을 적는다. 없거나 검색하지 않았으면 **`해당 없음`** 또는 **`테스트 미검색`**으로 명시한다.
- **추적 가능성**: 빈 절 없이 `해당 없음` 등으로 채웠는가(산출 템플릿 규칙).

---

아래 **헤딩 순서(`## 1.`~`## 6.`, 부록 A)**를 유지하고, 빈 절은 생략하지 말고 `해당 없음` 또는 한 줄로 처리한다.

```markdown

- **목적**: (이 범위에서 맡는 것 — 온보딩·분석 문서로 무엇을 빠르게 파악하게 할지)
- **접근**: 의존성(import·require·패키지)·설정·이벤트·콜백·I/O 경로를 따라 확장해 읽음. (스택에 맞게 조정)
- **Spec**: (확인한 PRD/TRD·설계 문서 경로·요약, 또는 `해당 없음`)
- **테스트**: (관련 테스트 파일 경로·역할 요약, 또는 `해당 없음` / `테스트 미검색`)
- **관련 문서**: (경로·역할 요약 링크 목록, 또는 `해당 없음`. 상세 시퀀스·계약 SSOT는 링크로 위임할 수 있다.)
- **비목적**(선택): 범위 밖을 한 줄~불릿으로만 적는다. 없으면 `해당 없음`.

## 1. 기능의 정의와 설명

(2~3문장) 이 모듈/기능이 **무엇을 하는지**, 시스템에서 **어떤 역할**인지 정의한다.

---

## 2. 디렉토리 구조

**역할**: 분석 범위의 **파일 지도**. 트리 + 파일·폴더별 **한 줄 역할** 주석.

```
<진입점 기준 상대 경로>/
├── entry.ext              # …
├── subdir/
│   └── …
```

- UI·Provider·훅·유틸 등 **역할별로 묶어** 표시한다.
- 범위 밖 파일은 트리에 넣지 않거나 `…`로 생략한다.

---

## 3. 진입 흐름

**역할**: **런타임에 코드가 어디로 들어오는지** — 라우트·CLI·이벤트·Job 진입, 상위 컴포넌트 → 하위 컴포넌트, **마운트/초기화 순서**.

- 트리형 코드 블록 또는 불릿으로 `진입 → 분기 → 주요 초기화 단계`를 정리한다.
- `useEffect`/이벤트 핸들러 등 **시간 순 초기화**가 있으면 번호를 붙인다.

---

## 4. 실행 시퀀스

**역할**: 진입 이후 **시간 순 실행** — 핵심 함수·API·상태 전이 한 줄씩.

- **트리형 코드 블록**으로 주요 파이프라인을 한눈에 보이게 한다(예: `startX()` 분기).
- **내부 Phase·상태·플래그**가 있으면 표로 정리한다.
- 필요 시 **Mermaid `sequenceDiagram`** 1개(§4 전용). 상세는 관련 문서 링크.
- **유저 관점 화면 전환**(라우트·모달 등)이 짧게 필요하면 이 절 끝에 불릿으로만 요약한다. 없으면 `해당 없음`.
```

## 완료 시 체크

- **DoD**: 위 **「분석 산출물의 Definition of Done」**을 만족하는가(Spec·테스트 명시, 코드 기준 서술).
- `## 분석 목적과 접근`, `## 1.`~`## 6.`, `부록 A`가 모두 채워졌는가.
- Mermaid가 렌더 규칙을 위반하지 않는가.
- **정적 의존** 다이어그램이 **§6에만** 있고 §5와 중복되지 않았는가.
- 진입점이 과대하지 않았는가. 과대했을 때 **범위를 나누었거나 사용자와 확정**했는가.
