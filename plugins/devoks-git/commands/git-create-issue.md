# git-create-issue

GitHub에 이슈를 등록한다. **이슈 본문을 채우기 전에** 아래 **태스크 분석 정리**를 먼저 수행한다. 사용자가 맥락만 주었으면 분석을 스스로 보완하고, 불명확하면 질문한다.

## 태스크 분석 정리 (필수, 이슈 작성 전)

다음 항목을 정리한다 (응답에 요약해 보여주거나, 내부적으로라도 빠짐없이 채운 뒤 이슈로 옮긴다).

1. **태스크 제목** — GitHub 이슈 **title**로 사용한다.
2. **배경 / 문제 정의** — 왜 필요한지, 현재 무엇이 잘못/부족한지.
3. **범위 (In scope)** — 이번에 할 일.
4. **범위 밖 (Out of scope)** — 이번에 하지 않는 일.
5. **영향 파일** — 예상 경로·모듈 (추정이면 `예상:` 표기).
6. **검증 시나리오** — QA/수동 확인 단계 (Given/When/Then 또는 체크리스트).
7. **미결정 사항** — 스펙 확정이 필요한 질문 (없으면 "없음").
8. **Labels (제안)** — 아래 **「Label / Assignees / Type」** 절 규칙으로 후보를 정한다.
9. **Assignees (제안)** — 담당자 GitHub 로그인 목록(없을 수 있음).
10. **Issue type (제안)** — 조직·저장소가 이슈 타입을 쓰는 경우에만 유효 값 후보(없을 수 있음).

분석이 끝난 뒤에만 `issue_write` / `gh` / 붙여넣기 본문을 만든다.

## Label / Assignees / Type

이슈 등록 시 **본문과 함께** 메타데이터를 채운다. **추론 → 검증 → 공백 시 질문** 순서를 따른다.

### Labels

- 태스크 **영향 경로·도메인·성격**으로 후보를 낸다 (예: `packages/auth` + 버그 → `auth`류, 결제 화면 → `payment`류, 신규 기능 → `enhancement`류 — **저장소 실제 라벨 이름과 일치해야 함**).
- 후보 각각에 대해 GitHub MCP **`get_label`**(`owner`, `repo`, `name`)로 **존재 여부**를 확인한다. 없는 이름은 **버리고** 사용자에게 "사용 가능한 라벨 중 선택" 또는 정확한 이름을 묻는다.
- 확신이 없거나 라벨 목록을 모르면: 분석 요약에 **제안 라벨**을 적은 뒤 **사용자에게 확인·수정**을 요청한다.
- **`issue_write`의 `labels`** 에는 검증된 문자열 배열만 넘긴다. 임의·추측 라벨로 API를 호출하지 않는다.

### Assignees

- 대화·태스크에 담당자가 명시되어 있으면 그 **GitHub 로그인**을 쓴다.
- 없으면 MCP **`get_me`**로 인증 사용자를 알고, **"본인에게 할당할지" 한 줄로 제안**한 뒤 사용자가 동의하면 `assignees`에 넣는다. 동의 없이 임의 할당하지 않는다.
- 담당자가 불명확하면 **`assignees` 생략** 또는 사용자에게 로그인(id) 입력을 받는다.

### Issue type

- GitHub **이슈 타입**은 org·repo 설정에 따라 없을 수 있다.
- MCP **`list_issue_types`**는 **조직 `owner`** 인자가 필요하다. 저장소가 org 소속이면 보통 `git remote`의 `owner`와 동일. 반환된 **허용 값만** `issue_write`의 `type`에 넣는다.
- 타입 미지원·목록 비어 있음·매칭 실패 시 **`type` 필드 생략**한다. 사용자에게 "타입 지정 필요 여부"를 물을 수 있다.

### 사용자 확인 (요약 표)

등록 직전 응답에 아래를 짧게 보여 주고, 비어 있거나 추론만 된 항목은 **질문으로 확정**한다.

| 항목 | 제안 값 | 확정 방법 |
|------|---------|-----------|
| Labels | (검증 통과한 이름들) | 사용자 수정 또는 확인 |
| Assignees | (로그인들 또는 비움) | 사용자 지정 / 본인 할당 동의 |
| Type | (유효 type 또는 비움) | `list_issue_types` 기준만 사용 |

## 이슈 본문 규격 (SSOT)

- **구조화된 작업 이슈**: `.github/ISSUE_TEMPLATE/feature_task.md` 섹션 순서를 따른다.

### 섹션 매핑 (분석 결과 → 본문)

`feature_task.md` 순서대로 채운다. **요약 표**는 어디로 가는지만 보여 주고, 규칙은 아래 **섹션별 지침**을 따른다.

| 분석에서 나온 것 | 채울 이슈 섹션 |
|------------------|----------------|
| 태스크 한 줄 요약 | `### Description` |
| 비즈니스·사용자 쪽 상위 목표 (짧게) | `### Objectives` |
| 구현 목표·완료 조건·검증 | `### Definition of Done` |
| 배경·범위·영향 파일 등 컨텍스트 | `### Additional Context` |
| 합의된 결정 / 미정 | `### Decisions (With Agents)` |

#### `### Description`

- 이슈 목록에 보이는 **1~3문장**만. 배경 장문은 넣지 않는다.

#### `### Objectives`

- **측정 가능한 상위 목표**만 불릿으로 짧게.
- 구현 디테일·완료 조건·QA 단계는 **Definition of Done**에만 둔다 (중복 금지).

#### `### Definition of Done`

- **구현 목표**: 불릿.
- **완료 조건**: `- [ ]` 체크박스.
- **검증 시나리오**: 단계 또는 체크리스트.
- 위 세 덩어리를 소제목(예: `**구현 목표**`)으로 구분해 읽기 쉽게 쓴다.

#### `### Additional Context`
- 아래 **고정 소제목**을 쓴다 (없는 항목은 "없음" 한 줄로 명시 가능).
  - `#### 배경 및 문제 정의`
  - `#### 범위`
  - `#### 범위 밖`
  - `#### 영향 파일(예상)`
- 필요 시 `#### 미결정 사항` 등 추가 소제목을 같은 섹션 안에 둔다.

#### `### Decisions (With Agents)`

- 합의된 것은 표 `| 항목 | 결정 |`에 행으로 넣는다.
- 미정은 결정 칸에 `미정` / `TBD`, 또는 분석에 미결정이 없으면 표에 **없음** 한 행.
- 미결정만 `Additional Context`에 둘 경우: `#### 미결정 사항` + **위와 같은 `>` 인용 본문**.

- **인용(Quote) 규칙 — 단일 블록**: `### Additional Context` **제목 줄만** 인용 밖에 둔다. 그 **다음 줄부터** 바로 아래 이슈 섹션인 다음 `### …`(예: `### Decisions (With Agents)`) **직전 줄까지** 들어가는 **모든 텍스트**(빈 줄, `####` 소제목, 불릿, 표, 문단)를 **하나의 연속 인용**으로 감싼다. 즉 해당 구간의 **줄마다** `>`로 시작한다 (빈 줄은 `>`만). 불릿은 `> - …`, 표는 각 데이터 행도 `> | … |`. **소제목별로 인용을 끊지 않는다.**

## 로컬 초안 (선택)

- 필요 시 `doc/issues/YYYY-MM-DD-slug.md`에 초안을 둘 수 있다.
- 첫 줄: `<!-- 제목: 제안 제목 -->`
- 본문: 위 **섹션 매핑**과 동일한 구조로 `### Description`부터 채운다 (`doc/issues/README.md`가 있으면 동일 규칙 참고).

## 저장소 식별

1. `git remote get-url origin` 실행.
2. `owner` / `repo` 파싱:
   - HTTPS: `https://github.com/<owner>/<repo>.git` 또는 `.../<owner>/<repo>` (`.git` 생략 가능)
   - SSH: `git@github.com:<owner>/<repo>.git`
3. 파싱 실패 시 사용자에게 owner/repo를 묻는다.

## 등록 우선순위

1. **GitHub MCP** `issue_write` — 호출 **전** 스키마 확인. `method: "create"`, `owner`, `repo`, `title`, `body` 필수. 확정된 경우에만 `labels`, `assignees`, `type` 배열/문자열 추가.
2. MCP를 쓸 수 없으면 **`gh issue create`** — `--label` (반복 또는 쉼표), `--assignee`, 이슈 타입을 CLI가 지원하면 해당 플래그(환경별 상이) 사용. 메타데이터를 못 넘기면 출력 본문 하단에 "수동으로 Label/Assignees/Type 지정" 안내.
3. 그도 안 되면 **웹 UI**용으로 제목·본문·제안 Label/Assignee/Type을 출력하고 붙여넣게 안내한다.

## 완료 시 산출물

- 생성된 이슈의 **URL**을 응답에 반드시 포함한다.
- 적용된 **Labels / Assignees / Issue type**을 한 줄로 요약한다 (없으면 "미지정").
- **태스크 분석 정리** 요약을 짧게 같이 보여주면 리뷰에 유리하다.
- 로컬 초안을 썼다면 이슈 번호·URL을 초안 파일 상단 주석에 남길지는 **선택**이다 (사용자 요청 시에만 편집).

## 금지·주의

- 사용자가 명시적으로 요청하지 않는 한 **`git commit` / `git push` 하지 않는다.**
- 이슈 본문에 **토큰·비밀번호·내부 키·개인정보**를 넣지 않는다.
