# Function Draft Implement (기능 구현)

## Overview

기능 명세를 받아 프로젝트 패턴에 맞게 기능을 설계·구현한다. UI 구현과 달리 **비즈니스 로직, API 연동, 상태 관리, 데이터 흐름**에 중점을 둔다. **요구 정리 → 설계·승인 → 브랜치·이슈 사전체크 → 구현 → 마무리** 순서를 따른다.

## 입력 파라미터

실행 시 아래 정보를 제공받는다.

| 파라미터 | 설명 | 예시 |
|----------|------|------|
| **1) 기능 명세** | 구현할 기능의 설명, 범위, 동작 방식 | `목록 CRUD, API 연동, 페이지네이션` |
| **2) 구현 추가정보** | 참고 파일, API 스키마, 제약 사항 등 | `목록 페이지 참고. /api/v1/items 사용` |

추가정보에 다음이 있으면 우선 반영한다.

- **참고 파일**: `@경로`, `~처럼`, `기존 OO 참고`
- **API/엔드포인트**: `API 연동`, `스키마명`, `openapi`
- **상태/저장**: `로컬 저장`, `캐시`, `오프라인`
- **제약**: `인증 필요`, `폼 검증`, `접근성`

## Steps

### 1. 요구사항 정리

- 기능 범위, 입력/출력, 에러 케이스 파악
- 기존 유사 기능 탐색 (`Grep`, `Glob`)
- API 스키마·계약·인증 방식 확인 (있는 경우)

### 2. 설계 및 사용자 확인

다음을 정리해 **승인을 받은 뒤** 코드 생성에 들어간다.

- 영향 받는 모듈·파일·경로
- 데이터 흐름 (API → 상태 → UI)
- 에러 처리·로딩·빈 상태 전략
- 신규 API·스키마·상수 정의

> 사용자 규칙에 "승인 후 코드 생성"이 있으면 이 단계를 생략하지 않는다.

### 3. 브랜치·이슈 사전체크

구현(코드 작성)에 들어가기 **직전**에, 현재 브랜치·연결 이슈를 확인하고 브랜치명·이슈 초안을 제안한 뒤 사용자 확인 후 적용한다(모드 `full`).

→ [`../skills/feature-workflow-runner/references/branch-issue-precheck.md`](../skills/feature-workflow-runner/references/branch-issue-precheck.md) (SSOT — 브랜치명 규칙·이슈 생성 위임·적용 범위 포함)

### 4. 구현

- 비즈니스 로직과 UI 분리 (훅, 유틸, Provider 등)
- 입력 검증·계약 준수
- 로딩·에러·빈 상태 UI 처리
- 프로젝트의 i18n 키 추가 규칙에 맞게 키 추가 (있는 경우)

### 5. 마무리

- 프로젝트의 린트 스크립트(예: `npm run lint`, `pnpm lint`)(Bash)로 린트 확인
- 미사용 import 제거
- 구현 요약 및 남은 작업(추후 API, 테스트 등) 안내

## 참조 룰 (Rules)

기능 구현 시 `.claude/rules/` 및 `.claude/refs/` 디렉터리의 규칙 파일들을 참조한다. **이 섹션에 내용을 복사하지 않고 원본 파일을 직접 읽어 적용한다.**

---

## Rules (기능 구현 시 준수)

- **승인 우선**: 구현 계획 후 사용자 승인을 받은 다음 코드 생성
- **패턴 우선**: 기존 코드 스타일·구조를 따르고, 신규 패턴 도입 시 일관성 유지
- **i18n**: 노출 문자열은 `t('key')`, `ko.json`/`en.json` 동시 추가
- **에러 경로**: 모든 비동기·API 호출에 에러 처리 및 사용자 안내
- **Git**: 브랜치·이슈 사전체크(Step 3)를 따른다. **커밋·푸시**는 사용자 명시 요청 시에만 수행한다.

- [ ] 로딩·에러·빈 상태 UI 처리
- [ ] i18n 키 추가 (`ko.json`, `en.json`)
- [ ] 린트 확인, 미사용 import 정리
- [ ] 구현 요약 및 추후 작업 안내
