# verify-requirements — 요구사항 체크리스트·구현 충실도 검증

기능 구현 전 **요구사항을 체크리스트로 구조화**하고, 구현 완료 후 **코드와 요구사항의 충실도를 체계적으로 검증**한다. 두 단계가 동일한 추적 ID(`[REQ-카테고리-번호]`)를 공유해 사전-사후가 연결된 하나의 검증 워크플로우를 구성한다.

---

## 범위 (중요)

**한다:**
- 요구사항 문서에서 항목을 추출해 추적 가능한 체크리스트 생성(checklist 모드).
- 구현 diff를 체크리스트와 대조해 구현/부분/미구현 분류 및 심각도별 갭 보고(verify 모드).

**안 한다:**
- 코드 자동 수정(갭 보고만, 보완 진행은 사용자 승인 후).
- 데이터 값 정합성 실측 → `devoks-sdlc:verify-data-flow`.
- 코드 품질·보안 리뷰 → `devoks-sdlc:code-review-diff-branch`, `devoks-sdlc:code-security-review`.
- 실제 앱을 조작한 실동작 확인(UI 조작+상태 영속) → `devoks-sdlc:verify-acceptance-test`.

---

## 호출 방법

```
/devoks-sdlc:verify-requirements [mode=checklist|verify] [spec=<요구사항 문서 경로>]
```

| 모드 | 시점 | 목적 |
|------|------|------|
| `mode=checklist` | 구현 시작 전 | 요구사항 추출 → 구현 목표 체크리스트 생성 |
| `mode=verify` (기본) | 구현 완료 후 | 체크리스트 + diff traceability → 커버리지 갭 보고 |

`spec` 로 요구사항 문서 경로를 직접 지정할 수 있다. 미지정 시 자동 탐색한다.

---

## 공통 Steps (두 모드 모두 실행)

### S1. 모드 판별

`mode=checklist` 면 **체크리스트 생성 모드**, `mode=verify` 또는 미지정이면 **구현 검증 모드**로 분기한다.

### S2. 요구사항 문서 탐색 및 로딩

1. **`spec` 인자 확인** — 지정된 경로를 요구사항 문서로 해석한다.
2. **자동 탐색** — 경로가 없으면 다음 위치를 순서대로 탐색한다:
   - `.claude/workspace/` — PRD, 워크플로우, 비즈니스 로직 SSOT
   - `.claude/plans/` — 구현 플랜 파일 (가장 최신 플랜 우선)
   - `doc/` — 기타 아키텍처/기능 문서
3. **브랜치명 키워드 매칭** — `git branch --show-current` 결과로 관련 문서를 좁힌다.
4. 탐색된 문서 목록을 출력하고, 추가/제외가 필요하면 사용자에게 한 줄 확인 후 진행한다.

### S3. 참조 문서 로딩

다음 문서를 읽고 검증 기준으로 적용한다:
- `.claude/refs/engineering-principles.md` — SSOT, Contract Validation, Fail-Fast 원칙
- `.claude/rules/agent-principles.md` — Spec Compliance, Fail Fast 정의
- `.claude/CLAUDE.md` — 도메인 용어, 아키텍처 개요

### S4. 요구사항 체크리스트 추출

요구사항 문서에서 다음 5개 카테고리로 항목을 추출하고 `[REQ-카테고리-번호]` ID를 부여한다.

| 카테고리 (약어) | 추출 대상 예시 |
|----------------|---------------|
| **기능 요구사항** (F) | 스텝 분기, 버튼 동작, 자동 진행 조건 |
| **비즈니스 로직** (B) | 임계값 계산식, PASS 판정 조건, 보정값 적용 순서 |
| **UI/UX 사양** (U) | 화면 레이아웃, i18n 키, 버튼 활성 조건, 색상 인디케이터 |
| **엣지케이스·에러처리** (E) | 최대 재시도 횟수, 에러 모달 분기, max cycle 처리 |
| **데이터·상태 계약** (D) | 저장 대상 키, 누적 방향, 동기화 조건 |

추적 ID 예시: `[REQ-F-01]`, `[REQ-B-03]`, `[REQ-U-02]`

## 체크리스트 생성 모드 전용 Steps (`mode=checklist`)

### C1. 체크리스트 출력

S4에서 추출한 항목을 카테고리별로 정렬하고 아래 **체크리스트 생성 템플릿**으로 출력한다.  
각 항목은 `[ ]` 체크박스 형식 — 구현자가 직접 체크하며 진행할 수 있도록 한다.

### C2. 체크리스트 파일 저장

체크리스트를 다음 경로에 파일로 저장한다:

```
.claude/workspace/checklists/<feature-name>.md
```

- `<feature-name>` 은 `$ARGUMENTS`의 요구사항문서를 확인하고 적절한 {feature-name}-{date} 형식으로 작성한다.
- `.claude/workspace/checklists/` 디렉토리가 없으면 생성한다.
- 파일이 이미 존재하면 덮어쓰기 전 사용자에게 한 줄 확인한다.

### C3. 구현 준비 확인

체크리스트 저장 후 사용자에게 다음을 확인한다:
- 누락된 항목이 있으면 추가 (파일도 함께 갱신)
- 범위에서 제외할 항목이 있으면 표시 (파일도 함께 갱신)
- 저장 경로를 안내하고 구현을 시작하도록 안내

---

## 구현 검증 모드 전용 Steps (기본)

### V0. 기존 체크리스트 파일 확인

`.claude/workspace/checklists/<feature-name>.md` 파일이 존재하는지 확인한다.

- **파일이 있으면** → 해당 파일을 로드해 `[REQ-*]` ID 목록을 재사용한다. S4(요구사항 재추출)를 건너뛴다.
- **파일이 없으면** → S4에서 요구사항 문서를 재추출해 ID를 새로 생성한다.

### V1. 브랜치 변경 범위 확보

1. `git log --oneline <base>..HEAD` — 커밋 목록 (변경 의도 파악)
2. `git diff <base>...HEAD --name-only` — 변경 파일 목록
3. 필요 시 `git diff <base>...HEAD` — 상세 diff (구현 위치 매핑용)

### V2. 구현 추적 (Traceability Mapping)

S4에서 추출한 각 요구사항 항목에 대해:
1. diff 및 변경 파일에서 **구현 위치(파일:라인)** 를 찾는다.
2. 다음 3가지 상태로 분류한다:
   - ✅ **구현 완료** — 요구사항과 코드가 일치
   - ⚠️ **부분 구현** — 일부만 구현되었거나 의도와 다르게 구현됨
   - ❌ **미구현** — 코드에서 대응 구현을 찾을 수 없음
3. 부분/미구현 항목은 아래 심각도 기준으로 판정한다.

### V3. 심각도 판정 기준

| 심각도 | 기준 |
|--------|------|
| 🔴 Critical | 핵심 비즈니스 로직 미구현, 데이터 손실/오염 위험, 보안 계약 위반 |
| 🟠 High | 주요 기능 요구사항 누락, 잘못된 알고리즘/수식 구현, 상태 기계 분기 오류 |
| 🟡 Medium | UI 사양 불일치, i18n 키 누락, 엣지 케이스 미처리, 부분적 구현 |
| 🔵 Low | 문서와 다른 변수명·주석, 로깅 누락, 미세 레이아웃 차이 |

### V4. 충실도 보고서 출력

아래 **구현 검증 템플릿**으로 작성한다.  
**템플릿에 포함된 이모지(📋 🧭 ✅ ⚠️ ❌ 📐 🔍 🔴 🟠 🟡 🔵 📊 🚀)를 반드시 그대로 포함하여 출력한다.**

### V5. 보완 진행 여부 확인

보고서 출력 후 사용자에게 반영 범위를 확인한다:
- 전체 반영 / High 이상 / Medium 이상 / Low 이상 4가지 선택지
- 확인이 필요한 사항이 있으면 먼저 질문하고 계획을 세운다.

## 참고

- **권장 워크플로우**:
  1. 구현 전 → `/devoks-sdlc:verify-requirements mode=checklist` — 구현 목표 체크리스트 확보
  2. 구현 후 → `/devoks-sdlc:verify-requirements mode=verify` — 커버리지 갭 검증
  3. 코드 품질 → `/devoks-sdlc:code-review-diff-branch` — 별개로 병행 실행
- **체크리스트 저장 위치**: `.claude/workspace/checklists/<feature-name>.md` — 브랜치명 그대로 파일명 사용
- **요구사항 문서 기본 위치**: `doc/workspace/` (PRD, 워크플로우), `.claude/plans/` (구현 플랜)
- **프로젝트 원칙**: `.claude/refs/engineering-principles.md`, `.claude/rules/agent-principles.md`
- **추적 ID 형식**: `[REQ-카테고리약어-번호]` — 예: `[REQ-F-01]` (기능), `[REQ-B-01]` (비즈니스로직), `[REQ-U-01]` (UI), `[REQ-E-01]` (엣지케이스), `[REQ-D-01]` (데이터계약)
- **ID 연속성**: `mode=checklist`에서 생성된 ID가 구현 검증 모드에서 동일하게 사용됨 — 사전/사후 추적 가능
