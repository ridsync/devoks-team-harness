# 브랜치·이슈 사전체크 (Precheck & Propose) — SSOT

> 구현 착수 **직전**(첫 코드 변경 전)에 현재 브랜치·연결 이슈 상태를 확인하고, 필요 시 브랜치명·이슈 초안을 **제안 → 사용자 확인 → 적용**한다.
> 이 규칙은 `feature-workflow-runner`(Phase 3), `feature-plan-executor`, `new-feature-draft`, `new-feature-github-issue`가 공유한다.

## 적용 시점

- **최초 1회** — 첫 구현 착수 직전(Phase 3 step 3, executor step 3, draft/github-issue step 3·4).
- Phase 1(FRD)·Phase 2(PLAN) 문서 작성 단계에서는 **적용하지 않는다**.

## 모드

| 모드 | 적용 대상 | 이슈 체크 |
|------|-----------|-----------|
| `full` | `feature-workflow-runner`, `feature-plan-executor`, `new-feature-draft` | 브랜치 + 이슈 모두 |
| `branch-only` | `new-feature-github-issue` | 이슈는 이미 SSOT로 주어짐 → **브랜치 제안·확인만** |

## 절차

### 1. 현재 브랜치 확인

- `git branch --show-current`(또는 저장소 관례에 맞는 동등 명령)으로 현재 브랜치를 조회한다.
- 원격 기본 브랜치(보통 `main`)와 비교해 **작업 브랜치 여부**를 판단한다.

### 2. 이슈 연결 확인 (`full` 모드만)

다음 **순서**로 연결 이슈를 탐색한다. GitHub 저장소를 검색하지 **않는다**.

1. `PLAN.md` frontmatter `issue:` (있으면)
2. `FRD.md` frontmatter `issue:` (있으면)
3. 대화·요청 컨텍스트 — 이슈 URL, `owner/repo#번호`, `#123` 형태
4. `new-feature-github-issue` 입력 이슈(해당 커맨드는 **이미 충족**으로 간주, `branch-only` 모드)

못 찾으면 **"연결 없음"**으로 간주한다.

### 3. 초안 제안

작업 성격·기능명·FRD/PLAN title(또는 draft/github-issue 입력)에서 다음을 **제안값**으로 만든다.

**브랜치명**

- **타입**: `feat`·`fix`·`docs`·`style`·`refactor`·`perf`·`test`·`build`·`ci`·`chore` 중 작업에 가장 가까운 것 — Conventional Commits, `devoks-git:git-commit-msg` 규칙.
- **이름 패턴**: `타입/간단-슬러그`(kebab-case). 연결 이슈 번호가 있으면 `타입/번호-슬러그`(예: `feat/128-partner-etc-page`).
- **생성 필요 여부**: 현재 브랜치가 원격 기본 브랜치(`main`)이면 **신규 생성 제안**. 이미 작업 브랜치면 **현재 브랜치 유지 제안**(새로 만들지 않음).

**이슈** (`full` 모드, 연결 없을 때만)

- **제목 초안**만 미리 제시한다(기능명·Goal 1문장 수준).
- 본문·Label·Assignee·Type 확정은 **`devoks-git:git-create-issue`** 절차에 위임한다(중복 정의 금지).

### 4. 상태 요약 · 사용자 확인

**둘 다 이미 충족**(작업 브랜치 + 연결 이슈)이면 한 줄 요약만 보여주고 **3→4 단계를 생략**하고 구현으로 진행한다.

그 외에는 아래 표 형식으로 요약하고 **한 번에** 확인을 받는다.

| 항목 | 현재 상태 | 제안 |
|------|-----------|------|
| 브랜치 | `<현재 브랜치명>` | `<유지 \| 신규: 타입/슬러그>` |
| 이슈 | `<URL/번호 \| 연결 없음>` | `<생성 제안: 제목 초안 \| 해당 없음>` |

**선택지**(사용자에게 직접 제시):

- (A) 브랜치만 적용
- (B) 이슈만 생성 (`full` 모드, 연결 없을 때)
- (C) 둘 다 적용
- (D) 둘 다 건너뛰고 계속

- 이슈 생성은 **선택 사항** — 거부해도 구현을 **차단하지 않는다**.
- 브랜치도 (D) 선택 시 현재 브랜치에서 계속(원격 기본 브랜치에서 작업 중임을 한 줄로 알림).

### 5. 적용

승인된 항목만 실행한다.

**브랜치 (A 또는 C, 신규 생성 승인 시)**

- 원격 기본 브랜치 최신 기준으로 분기(`git checkout -b …` 등 저장소 관례).
- **범위**: 생성·체크아웃까지만. **커밋·푸시·PR은 하지 않는다.**

**이슈 (B 또는 C, `full` 모드)**

- `devoks-git:git-create-issue` 절차를 따른다(태스크 분석 → 본문·메타 확정 → 등록).
- 생성된 **URL/번호**를 기록:
  - FRD/PLAN 워크스페이스가 있으면 해당 문서 frontmatter `issue:` 필드에 반영.
  - 이슈가 **먼저** 생성되고 브랜치도 (C)로 승인됐으면, 브랜치명에 이슈 번호를 반영한 뒤 생성한다.

**`branch-only` 모드 (`new-feature-github-issue`)**

- 입력 이슈 번호를 브랜치명에 반영(`타입/이슈번호-슬러그`).
- 이슈 생성 단계는 **수행하지 않는다**.

## 금지·주의

- GitHub 이슈 목록/유사 이슈 **자동 검색 금지** — 컨텍스트·frontmatter·사용자 입력만 사용.
- 사용자 확인 없이 브랜치 생성·이슈 등록 **금지**.
- 이슈·PR·채팅에 **토큰·비밀번호·내부 키·개인정보** 넣지 않는다.
