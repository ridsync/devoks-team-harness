# 태스크 구현 위임 — code-implementer 프로토콜

> Phase 3(태스크 실행)의 구현·검증은 **`code-implementer` 에이전트 위임이 기본 경로**다.
> 목적은 컨텍스트 격리 — 구현 중 파일 다독·테스트/린트 출력이 메인 대화에 적재되지 않게 하고,
> 메인 루프는 오케스트레이션(라우팅·리포트 판정·PLAN 갱신·사용자 확인)에 집중한다.

## 라우팅 — 위임 vs 직접 (실행 측 SSOT)

실행 시점의 즉흥 판단을 없앤다. Task의 `size` 마커(부여 기준 SSOT: `task-pr-splitting.md`)를 기계적으로 적용한다.
위임 적합성의 스펙 완결성 판단은 `docs/plugin-management.md` §12 3기준 #1과 아래 공통 예외 ③에 반영된다.

| 신호 | 라우팅 |
|------|------|
| `size: S` | 메인 루프 **직접 구현** — 위임 오버헤드(기동·재독·리포트)가 본작업보다 큼 |
| `size: M` | `code-implementer` **위임** (기본 경로) |
| `size: L` | 위임 전 **분해 재검토** — 쪼갤 수 있으면 PLAN 갱신 후 재라우팅, 못 쪼개면 메인 루프 직접 |
| 마커 없음 (구버전 PLAN) | `M`으로 간주 → 위임 |
| 공통 예외 ① — 에이전트 tools(`Read, Grep, Glob, Bash, Edit, Write`) 밖 도구(예: Figma·CodeGraph MCP) 필요 | 메인 루프 직접 |
| 공통 예외 ② — 사용자가 직접 실행을 명시 요청 | 메인 루프 직접 |
| 공통 예외 ③ — **스펙 미완결**(§12 3기준 #1 위반) — `file:` 미정·`traces:` AC/CTR/EDGE 본문이 FRD에 없음·구현 방식이 §1 Approach와 충돌·blocked 해소 결정이 아직 `context`에 없음 | **`size`와 무관하게** 메인 루프 직접 — 사용자 확인·스펙 확정 후 재라우팅. 위임 전에 스펙을 닫는다 |

- 직접 구현 시에도 절차(읽기 전 변경 금지·재사용 우선·검증 후 `[x]`·인계 노트 기록)는 동일하다.
- **런타임 보정**: `M`으로 위임했는데 실제 스코프가 `L`이었다면 에이전트가 blocked(스코프 초과)로 반환하고, 메인 루프가 인수하거나 재분해한다.

### 위임 손익분기 (비용·이득)

위임의 순이득은 **메인 컨텍스트 윈도우 절약**이다. `code-implementer`는 `model: sonnet` 고정이므로, 세션 모델이 opus/fable이면 위임이 **시간·rate limit 절약**도 함께 준다(plan-Opus/execute-Sonnet 패턴). 다만 cold-start 재독 오버헤드는 세션 모델과 무관하게 존재한다.

| 상황 | 권장 |
|------|------|
| **현재 시점(재개 포함) 기준** 잔여 미완 Task **≤ 3개**이거나 **단일 모듈·단일 PR**에 집중 | `size: S`를 넉넉히 부여해 직접 구현 비중을 높인다 — 위임 기동·재독 오버헤드가 이득을 상쇄하기 쉽다. PLAN 작성 시점 총량이 아니라 실행 시점(세션 재개 포함)마다 재평가 |
| PLAN이 **장기**(미완 Task 다수·여러 PR·반복 재독) | `M` 위임 이득이 큼 — 메인 컨텍스트 오염 방지 + (세션이 opus/fable이면) rate limit 절약이 함께 작동 |
| 세션 모델이 opus/fable이고 태스크당 재독 범위가 넓음 | `S` 임계를 보수적으로(함수 1~2개·출력 작음) 유지 — 재독 오버헤드는 여전히 존재하므로 `task-pr-splitting.md` 부여 기준 참고 |

## 입력 계약

메인 루프(호출자)가 아래 파라미터를 에이전트 프롬프트에 주입한다.

| 파라미터 | 필수 | 내용 |
|---------|------|------|
| `task` | ✅ | PLAN.md의 해당 Task 블록 **원문 그대로** (TASK-ID·설명·`file:`·`traces:` 포함) |
| `plan` | ✅ | PLAN.md 절대 경로 — §1 Approach·§4 의존성 참조용 (필요 시 에이전트가 직접 Read) |
| `frd` | ✅ | FRD.md 절대 경로 — `traces:`의 AC/CTR/EDGE 본문 확인, 테스트 설명에 AC ID를 박는 계약 이행용 |
| `context` | 선택 | 메인 루프가 아는 힌트 — **인계 노트(윈도잉 적용, 아래)**·재사용할 유틸/패턴 경로·blocked 해소 결정·세션에서 합의된 제약 |

## 리포트 스키마

에이전트의 최종 메시지는 아래 둘 중 하나다. 메인 루프는 리포트만 근거로 판정한다(스키마 SSOT는 이 파일, `code-implementer` 에이전트는 이를 미러).

**`status: done`**
- TASK-ID, 변경 파일 목록(신규/수정 구분)
- **검증 근거(필수)** — 아래 네 가지를 모두 포함. 요약만 있고 원문 조각이 없으면 done으로 취급하지 않는다.
  1. **실행 명령** — 셸에 입력한 문자열 그대로
  2. **종료코드** — `0`(또는 성공으로 간주하는 exit code) 명시
  3. **출력 tail 1줄 원문** — 테스트/린트 프레임워크의 pass·fail·에러 **카운트 또는 최종 결과 줄**을 도구 출력에서 그대로 복사(재서술 금지). 예: `Tests: 3 passed, 3 total` / `✓ 12 tests passed` / `error TS2345: ...`
  4. **로직 Task 추가** — 추가·변경한 테스트 파일 경로 + 해당 케이스명(또는 `it`/`test` 설명) 1개 이상
- **인계 노트(필수)** — 만든 헬퍼/유틸, 발견한 제약, 다음 태스크에 영향 주는 결정 1~3줄. 없으면 "없음".
- Task `file:` 밖 파일을 변경했다면 사유 명시

> 메인 루프는 전체 로그를 보지 않는다. tail 1줄 원문 + 종료코드가 **재현·위조 난도를 높이는 최소 신뢰 경계**다. 의심 시 메인 루프가 동일 명령을 직접 재실행해 확인한다.
>
> **재실행 트리거(하나라도 해당하면 의심 없이 재실행)**
> - exit code와 tail 원문의 의미가 불일치(예: exit 0인데 tail에 `fail`/`error`/`✗` 포함, 비0인데 `pass`류 표현)
> - tail 원문이 표준 요약 포맷이 아니거나 정보가 없음(빈 줄, 명령 echo만 있음 등)
> - 실행 명령이 Task `file:` 범위보다 뚜렷이 넓음(예: 특정 파일 대상 대신 전체 스위트 실행 — 범위 밖 통과로 착시 가능)
> - 로직 Task인데 "로직 Task 추가"(테스트 파일·케이스명) 항목이 비었거나 기존 테스트 재사용만 언급
> - 같은 PLAN 내 직전 done 리포트들과 명령 패턴이 이유 없이 달라짐

**`status: blocked`**
- TASK-ID, 시도한 것
- 정확한 blocker — 스펙 공백(무엇이 미정인지) / 테스트 실패 요약 / 코드·설계 충돌
- **부분 변경 파일 목록과 상태(은폐 금지)** — 롤백 필요 여부 포함
- 해소 제안 1~2개

## 에스컬레이션 (blocked 처리)

1. **스펙 문제**(요구 미정·설계 공백) → 메인 루프가 사용자에게 확인하고, 결정을 `context`에 덧붙여 **재위임**한다.
2. **기계적 문제**(환경·의존성·선행 태스크 결함) → 메인 루프가 직접 해소한 뒤 재위임한다.
3. blocked 태스크는 `[x]`로 바꾸지 않고 막힌 이유를 기록한다. → `progress-tracking.md` 부분 완료 금지

## 순차 위임 원칙

**한 번에 한 태스크만** 위임한다. `[P]` 마커는 "지금 시작해도 된다"는 선택 순서의 자유일 뿐,
동시 fan-out 지시가 아니다(같은 워킹트리에 병렬 Edit 시 파일 충돌).

## 인계 노트 누적 — 크로스 태스크 지식 유지

cold-start 위임의 지식 단절(같은 모듈 재탐색·네이밍/패턴 drift)을 메우는 채널이다.

- 메인 루프는 각 done 리포트의 **인계 노트**를 수집해, 같은 PLAN 실행 내 **다음 위임의 `context`에 주입**한다.
- 직접 구현한 태스크(S·공통 예외)도 같은 기준으로 인계 노트를 남겨 다음 위임에 잇는다.
- 인계 노트는 요약 채널이다 — 세부가 필요하면 구현자가 `plan`·`frd`·코드를 직접 읽는다.

### 윈도잉 (누적 상한)

무제한 누적은 위임 `context`를 비대하게 해 격리 이득을 상쇄한다.

- **기본**: 현재 착수 중인 **PR 그룹** 내 done 태스크의 인계 노트만 포함한다.
- PR 그룹을 넘는 선행 지식이 필요하면, 그룹 밖에서 **가장 최근 done 3개**까지만 추가한다.
- **총량 상한**: `context` 인계 노트 블록은 **20줄 이하**. 초과 시 오래된 항목부터 제거하고, 제거된 요약이 필요하면 `plan`·코드를 재독한다.
- roadmap의 continuation(P/R그룹 단위 세션 유지) 도입 시 이 윈도잉은 상위 호환으로 완화된다.

## PLAN.md 갱신 주체 — 메인 루프 단독

에이전트는 **PLAN.md·FRD.md·워크스페이스 문서를 수정하지 않는다.**
체크박스 `[x]`·frontmatter `status` 전환은 done 리포트의 검증 근거(명령·종료코드·tail 1줄 원문)를 확인한 뒤 **메인 루프가** 수행한다. → `progress-tracking.md`
