# test-run-triage — 테스트 실행·실패 triage

테스트 스위트를 실행하고 실패를 **분류·원인 추정·수정안**으로 정리한다.
코드 수정은 제안만 하고, 적용은 사용자 승인 후에 한다.

---

## 범위 (중요)

**한다:**
- 전체 또는 지정 범위 테스트 실행.
- 실패를 회귀 / 플래키 / 스펙 불일치 / 환경 4범주로 분류.
- 각 실패의 근본 원인 추정 + 수정안 제시(위치 포함).

**안 한다:**
- 코드·테스트 자동 수정(제안만, 적용은 승인 후).
- 새 테스트 작성 → `devoks-sdlc:test-author`.

---

## 호출 방법

```
/devoks-sdlc:test-run-triage [scope=<경로/패턴>]
```

- `scope` 미지정 시 전체 스위트. 변경 범위만 빠르게 보려면 scope로 좁힌다.

---

## 절차

### 1. 실행은 context-mode로

테스트 출력은 대용량이라 메인 컨텍스트를 잠식한다.
- 프로젝트의 테스트 명령을 감지한 뒤 `ctx_batch_execute` / `ctx_execute`로 실행한다.
- 전체 원문이 아니라 **실패·에러·요약만** 메인 컨텍스트로 surface한다(원문 바이트는 샌드박스 격리 — 이 repo의 context-mode 정책 준수).

### 2. 실패 분류

각 실패를 다음 범주로 태깅한다.

| 범주 | 신호 |
|------|------|
| 회귀(regression) | 최근 변경으로 깨짐, 재현 일관적 |
| 플래키(flaky) | 재실행 시 통과/실패 변동, 타이밍·순서 의존 |
| 스펙 불일치 | 테스트 기대값이 변경된 의도된 동작과 어긋남 |
| 환경(environment) | 의존성·설정·외부 자원 부재로 실패 |

### 3. 근본 원인 추정 (증거 기반)

- 관련 코드를 codegraph/Read로 좁혀 확인한다. **추측 금지** — 로그·diff·심볼 근거로만 판단.
- 플래키 의심 시 재실행으로 변동성을 확인한다. 원인 후보는 `../test-author/references/test-quality-bar.md` §3(결정성 — 시간/난수/동시성/네트워크·FS·DB/순서)을 1차 체크리스트로 쓴다.

### 4. 리포트

아래 템플릿으로 출력한다. 이모지를 그대로 포함한다.

```markdown
## 📋 실행 대상
- 범위: [전체 / scope]
- 결과: 통과 N · 실패 M · 스킵 K

## 📊 실패 요약
| # | 테스트 | 범주 | 심각도 | 위치 |
|---|--------|------|--------|------|
| 1 | ... | 회귀/플래키/스펙/환경 | 🔴/🟠/🟡/🔵 | `file:line` |

## 🔍 실패 상세
### 1. [테스트명]
- 범주:
- 원인 추정:
- 수정안:
- 위치: `file:line`

## 🚀 다음 액션
- 즉시:
- 후속:
```

## 완료 기준

- 실패 0이면 "통과"를 명시적으로 선언한다.
- 실패 시 각 항목에 범주 · 원인 추정 · 수정안 · 위치가 채워져 있다.
- 코드 수정은 사용자 승인 후에만 진행한다.
