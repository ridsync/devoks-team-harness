# 검증 리포트 템플릿 (Phase 5)

이번 검증 사이클의 결과를 표준 형식으로 남기는 템플릿이다. 저장 위치는 프로젝트 문서 디렉토리(`doc/` 등),
파일명은 `<대상>-data-verification-result.md` 권장.

리포트는 다음 셋을 동시에 만족해야 한다: **빠르게 결론을 읽을 수 있다 / 증거로 결론을 검증할 수 있다 / 다음 작업으로 이어진다.**

---

## 템플릿 본문 (그대로 복사해 채워 쓴다)

```markdown

- 대상 브랜치/리비전: `<branch or commit>`
- 검증 대상: `<TargetData>` (인벤토리 N건)
- 환경: `<Environment>` (browser / server / db / cli)
- 수집 방식: `<CollectionMethod>` — <짧은 한 줄>
- 실행 시나리오: `<TriggerFlow>`
- 실행 시각: `<YYYY-MM-DD HH:MM ~ HH:MM>`
- 기준 문서: `<SpecSource>`
- 임시 변경/게이트 (있다면): `<예: '2세트 후 PASS' 검증용 게이트>`

## 종합 판정: ✅ 정합 / ⚠️ 일부 관찰 / ❌ 불일치

<한 단락으로 결론 요약. 무엇이 일치했고, 어떤 관찰/불일치가 있었는지.>

## 단계별 대조

각 대상 값에 대해 **의도값(L_apply) ↔ 저장값(L_persist) ↔ 재로드값(L_reload)** 의 3중 대조 결과.

| 대상 값 | 단계/이벤트(증거) | 의도값 | 저장값 | 재로드값 | 판정 |
|--------|------------------|--------|--------|----------|------|
| <field> | `apply.<x>` / `persist.<store>` / `final.snapshot` | <값> | <값> | <값> | ✓ / ⚠ / ✗ |

> 증거 컬럼은 이벤트명·msgid·로그 라인 등 **원시 출처로 역추적 가능한 식별자**를 적는다.

## 기준 대조 (정확성)

`<SpecSource>` 의 공식·임계값·스키마를 직접 재현해 비교한 결과.

- **공식**: `<공식을 한 줄 수식으로>`
- **재현 계산**: 입력 `[…]` → 산출 `<값>` → 기준값 `<값>` → 차이 `<값>` → verdict `<PASS/APPLY/...>`
- **임계값 분기**: 어떤 분기로 갔는지, 워크플로우 문서의 어느 절과 일치하는지.

여러 값이면 표로 정리한다.

## 발견 사항

심각도(severity)와 원인 추정/권장 조치를 함께 적는다. 정합성과 별개의 관찰도 별도 표기.

- **[Critical]** <불일치/버그>: <증거 출처> — 원인: <…>, 조치: <…>
- **[Major]** …
- **[Minor]** …
- **[Observation]** <정합성과 별개의 환경/누적 등 관찰> — 권장: <…>
```

## 작성 팁

- **결론이 먼저, 증거가 나중**. 종합 판정 한 줄을 읽으면 의사결정이 가능해야 한다.
- **표는 좁고 깊게**. 한 행이 한 사실(대상 값)을 말하고, 컬럼은 동일 차원으로만.
- **수치는 단위와 함께**. `0.50101` 보다 `overrideGrindLeftOffset = 0.50101` 같이 의미 부여.
- **"같다"는 같음을 증명해야 한다** — 의도값·저장값·재로드값 셋이 모두 적혀 있어야 일치 판정이 검증 가능하다.
- **발견 사항과 미검증을 분리**. 발견 사항은 "본 것", 미검증은 "보지 못한 것" — 둘이 섞이면 안전성 평가가 흐려진다.
