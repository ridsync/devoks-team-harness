# 설계 스펙 (Design Spec) — FRD §4 작성 가이드

> FRD가 **무엇을(요구)** 고정한다면 §4는 **어떻게(구조)** 를 고정한다.
> 컴포넌트·패턴·모듈 배치를 착수 전에 정해, PLAN의 Task 분해가 임의 구조로 흩어지지 않게 한다.
> 요구 누락은 **묻는다**(추측 금지). 설계 공백은 **합리적 안을 먼저 제안한 뒤 확인**한다 — 이 차이가 이 문서의 핵심이다.

## 언제 얼마나 쓰나 — 복잡도 임계

§4를 어디까지 채울지는 기능 복잡도로 가른다.

| 복잡도 | 처리 |
|--------|------|
| **단순** (단일 파일·기존 패턴 그대로·새 구조 없음) | 4.1만 채우고 4.2~4.4는 1~2줄 또는 생략 |
| **임계 초과** (아래 신호 중 하나 이상) | 4.2~4.5를 채우고, 빈 설계는 **제안 → 확인** 절차로 채운다 |

임계 신호(`.claude/rules/agent-principles.md` "Ask Before Expanding Scope"와 동일 기준):
- 영향 파일 **3개 초과**
- **신규 모듈/계층/상태 컨테이너** 도입
- **아키텍처/계약 변경** (의존 구조, 데이터 흐름, 공개 API)
- **새 디자인 패턴** 도입 (프로젝트에 선례 없는 구조)

## 능동 제안 절차 (임계 초과 시)

요구 누락의 "묻기만"과 구분되는, 설계 전용 흐름이다.

1. **패턴 탐색** — 코드베이스에서 유사 기능의 기존 상태관리·훅·유틸·모델 패턴을 먼저 찾는다(재사용 우선, 새 구조 창작 최소화).
2. **후보안 도출** — 탐색 결과를 근거로 **설계안 1개**(트레이드오프가 크면 대안 1개 동봉)를 만든다.
3. **제안 → 확인** — "이렇게 설계하려는데 맞는지" 한 번에 모아 사용자에게 확인한다. 빈칸을 그냥 "채워달라"고 떠넘기지 않는다.
4. **확정 → §4 작성** — 확인된 안으로 4.2~4.5와 DSN 표를 채운다.

> 요구(REQ/AC)는 사용자만 정할 수 있으니 **묻는다**. 설계는 코드 근거로 **에이전트가 먼저 제안**할 수 있으니 제안한다. 둘을 섞지 않는다.

## §4 하위 섹션 작성법

### 4.1 화면/흐름
라우팅·네비게이션 단위, 조건부 렌더링/분기, 시안(§6.3 Assets ID 참조). UI가 없는 기능이면 데이터 흐름으로 대체.

### 4.2 컴포넌트 구조
컴포넌트 트리 + 각 컴포넌트 **책임** + **props/상태 경계**(상태를 누가 소유하는가).
- 좋은 예: "필터 상태는 부모가 소유, 자식은 값+콜백만 받는다" — 소유 위치가 명시됨.
- 나쁜 예: "필터 컴포넌트 만들기" — 책임·경계가 없어 PLAN에서 임의 분해됨.

### 4.3 적용 디자인 패턴 (DSN)
프로젝트의 **확립된 패턴**에 매핑한 설계 결정을 `DSN-xxx` 표로. 새 패턴을 발명하지 말고 기존 것을 가리킨다.
도메인 중립 결정 예: 상태관리 패턴 재사용 / 계산 로직을 순수 함수로 분리 / 모델 클래스+직렬화 / 계약 검증 헬퍼 / 데이터 저장소 선택.

### 4.4 모듈/레이어 배치
신규·수정 파일이 들어갈 디렉토리, 기존 모듈/계층 구조와의 관계, 저장소/계층 선택과 그 근거.

### 4.5 재사용 대상
기존 훅/유틸/컴포넌트(§6.1 참고 코드와 교차 참조)와 기존 로직 변경 최소화 방침.

## DSN ID 규칙

- **부여 단위** — 하나의 *설계 결정*마다 `DSN-001`, `DSN-002` … (컴포넌트 구조/패턴/배치 결정).
- **traces 참조** — PLAN Task의 `traces:`에 `DSN-xxx`를 인용해 "이 Task가 어떤 설계 결정을 구현하는지" 역추적할 수 있다.
- **커버리지 강제 아님** — `AC/CTR/EDGE`는 **누락 0**(모두 어떤 Task가 커버)이 강제다. **DSN은 다르다**: 설계 결정은 구조 선택이라 Task와 1:1로 떨어지지 않으므로, 참조는 권장하되 누락 0 점검 대상이 **아니다**. → `references/traceability.md`

## 프로젝트 패턴 출처

특정 프로젝트의 패턴 카탈로그는 이 문서에 하드코딩하지 않는다. 작업 중인 저장소에 규칙/가이드 파일이 있으면 **직접 읽어 따른다**:
`.claude/rules/`(코딩·아키텍처 규칙) · `CLAUDE.md`(구조·상태관리·데이터 저장소 결정) · `DESIGN.md`(UI 계약) 등. 없으면 코드베이스의 기존 패턴을 탐색해 근거로 삼는다.
